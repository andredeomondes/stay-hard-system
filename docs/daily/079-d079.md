# Dia 79 — Transaction e Isolation

## Conceito do Dia

**Transactions** garantem consistência de dados. Se algo falha no meio, tudo volta ao estado anterior (rollback).

## @Transactional básico

```java
@Service
@RequiredArgsConstructor
public class TarefaService {
    
    private final TarefaRepository repository;
    private final ProjetoRepository projetoRepo;
    
    @Transactional
    public Tarefa criarTarefa(CriarTarefaRequest request) {
        // Se qualquer exception for lançada, faz rollback
        Tarefa tarefa = new Tarefa();
        tarefa.setNome(request.nome());
        tarefa = repository.save(tarefa);
        
        // Simula erro
        if (request.nome().isEmpty()) {
            throw new RuntimeException("Nome inválido!");
        }
        
        return tarefa;
    }
    
    @Transactional
    public void moverTarefa(Long tarefaId, Long novoProjetoId) {
        Tarefa tarefa = repository.findById(tarefaId)
            .orElseThrow(() -> new RuntimeException("Não encontrada"));
        
        Projeto projeto = projetoRepo.findById(novoProjetoId)
            .orElseThrow(() -> new RuntimeException("Projeto não encontrado"));
        
        tarefa.setProjeto(projeto);
        repository.save(tarefa);
        // Se falhar, ambos voltam ao estado anterior
    }
}
```

## Propagation

```java
@Transactional(propagation = Propagation.REQUIRED) // padrão
// REQUIRED: usa transação existente ou cria nova
// REQUIRES_NEW: sempre cria nova (suspende a existente)
// NESTED: cria savepoint (só PostgreSQL)
// NEVER: erro se houver transação
```

## Isolation Levels

```java
@Transactional(isolation = Isolation.READ_COMMITTED) // padrão para PostgreSQL
// READ_UNCOMMITTED: lê dados não commitados (sujo)
// READ_COMMITTED: só lê commitados
// REPEATABLE_READ: mesma query = mesmo resultado
// SERIALIZABLE: mais lento, mais seguro
```
