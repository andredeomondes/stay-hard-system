# Dia 200 — Continuous Improvement: Retrospectives

## Conceito do Dia

Retrospectivas melhoram continuamente o time e o produto.

---

## Formato: Start, Stop, Continue

```
START: O que devemos começar a fazer?
STOP: O que devemos parar de fazer?
CONTINUE: O que está funcionando bem e devemos continuar?
```

## Exemplo de Retrospectiva

```markdown
# Retrospectiva Sprint 4

**Data:** 2026-03-29
**Participantes:** 5
**Format:** Start, Stop, Continue

## Start
- [ ] Pair programming para tasks complexas
- [ ] Code review antes de merge
- [ ] Documentar decisões de arquitetura

## Stop
- [ ] Deployar sextas-feiras (moratório)
- [ ] Ignorar warnings de linter
- [ ] Reuniões sem pauta definida

## Continue
- [x] Daily standups curtos (15 min)
- [x] Testes unitários (>80% coverage)
- [x] Revisão de PRs em < 24h
- [x] Documentação no Confluence

## Action Items
| Ação | Responsável | Prazo |
|------|-------------|-------|
| Configurar merge blocking no GitHub | DevOps | 2026-04-01 |
| Criar template de PR | Tech Lead | 2026-04-03 |
| Adicionar pré-commit hook | DevOps | 2026-04-05 |
```

## 5 Whys (Root Cause Analysis)

```markdown
## Problema: Deploy atrasou 2 dias

Why 1: Feature não ficou pronta a tempo
Why 2: Task era maior do que estimado
Why 3: Não quebramos em tasks menores
Why 4: Pressão para entregar rápido
Why 5: Sprint planning não considerou buffer

**Root Cause:**Planning não tinha buffer para imprevistos

**Action:**Adicionar 20% de buffer no sprint planning
```
