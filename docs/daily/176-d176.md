# Dia 176 — System Design: Como Projetar

## Conceito do Dia

System design avalia capacidade de arquitetar sistemas escaláveis.

---

## Passos

```
1. Clarify Requirements
2. Define Scope
3. High-Level Design
4. Deep Dive em componentes
5. Trade-offs e alternativas
```

## Exemplo: Design Twitter

### Requirements

```
Functionais:
- Postar tweets (140 chars)
- Feed personalizado
- Seguir usuários
- Search

Não-funcionais:
- 300M usuários
- 500M tweets/dia
- Alta disponibilidade
- Baixa latência no feed
```

### High-Level Architecture

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│ Clients │────▶│   CDN   │────▶│  Cache  │
└─────────┘     └─────────┘     └────┬────┘
                                     │
                                     ▼
┌─────────┐     ┌─────────┐     ┌─────────┐
│  Write  │────▶│ Message │────▶│   DB    │
│  Path   │     │  Queue  │     │ Sharded │
└─────────┘     └─────────┘     └─────────┘
```

### Componentes

| Componente | Solução |
|------------|---------|
| Tweets Storage | PostgreSQL (sharded) ou Cassandra |
| Timeline | Redis (Fan-out) |
| Search | Elasticsearch |
| Media | S3 + CloudFront |
| Metrics | Kafka + Druid |

---

## Perguntas a Fazer

```
- Quantos usuários esperados?
- Qual o read/write ratio?
- Quais as constraints de latência?
- Quão disponível precisa ser?
- Qual budget/opinião técnica?
```
